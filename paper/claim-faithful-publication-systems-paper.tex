\begin{abstract}
This document explains how one public mathematics repository is organised.
The repository contains formal statements and proofs written in Lean, a
program that checks proofs.  It also contains papers, a front page, and other
files that explain those proofs to people.  Lean checks the formal mathematics;
it does not read those explanations.  The repository therefore adds a reviewed
claim record between the proofs and the public documents.  That record says
what selected results mean, what evidence supports them, and what remains
open.  Programs build navigation files from the record and stop a release when
named relationships no longer agree.  GitHub runs the formal proof check and
the public-release checks as separate jobs.  A mathematician still decides
whether the formal statement is the intended one and whether the public words
describe it faithfully.  The architecture makes those reviewed decisions
repeatable; it does not replace them.  This paper names the actual files,
shows the complete flow, follows one result from Lean source to a public page,
and states exactly where the automatic guarantee ends.
\end{abstract}

\section{What this repository is for}\label{sec:purpose}

This is a public collection of files about two unsolved problems in
mathematics, Erd\H{o}s Problems 249 and 257.  Both problems remain open.  The
project proves intermediate results, exact reformulations, and finite examples
around them.  It does not claim to solve either problem.

The collection has two jobs:

\begin{enumerate}
\item It stores precise mathematical statements and proofs in \emph{Lean}.
  Lean is a proof assistant: a program that checks whether a formal proof
  establishes the exact formal statement written for it.
\item It explains that work to people through a README, a scope statement,
  papers, source maps, and data that other programs can read.
\end{enumerate}

A \emph{repository} is simply the versioned collection containing those
files.  A \emph{public claim} is a statement to readers about what the project
has proved or has not proved.  A \emph{release} is one fixed version of the
files offered to readers.  No knowledge of Lean, Git, number theory, or the
project's development history is assumed in this document.

The central problem is easy to state.  A Lean proof can remain correct while a
paper or README describes it incorrectly.  For example, a proof covering a
finite list of inputs can be described accidentally as a result covering all
inputs.  Lean cannot catch that wording error because the README is not part of
the formal proof.

The repository addresses this gap with five visible parts:

\begin{enumerate}
\item \textbf{Lean source} containing formal statements and proofs;
\item a \textbf{human-reviewed claim record} describing selected results and
  their limits;
\item \textbf{authored public documents} such as the README and papers;
\item \textbf{generated indexes and summaries} that reorganise recorded
  information; and
\item a \textbf{release program and continuous-integration workflow} that run
  the recorded comparisons before publication.
\end{enumerate}

The shortest accurate summary is:

\begin{quote}
Lean checks the formal proofs.  Mathematicians review what those proofs mean
and how they may be described.  The release machinery checks that the reviewed
relationships remain intact before publication.
\end{quote}

Some boundaries run through a file rather than around it.  For example, most
of the README is authored prose, while marked blocks inside it are written by
a builder.  The claim file is reviewed data, while scripts refresh source-line
coordinates and a module graph inside it.  The rule is therefore not ``JSON is
authored'' or ``Markdown is generated.''  Each generated region names its
builder.  A stale generated region is rebuilt by that owner, not repaired by
hand.

The generated views do not create new mathematics.  They select and reorganise
information so that a person or another program can enter the repository
without reading every source file.  Examples include:

\begin{itemize}
\item \texttt{scripts/build\_corpus\_descriptor.py}, which writes the human and
  machine orientation views;
\item \texttt{scripts/build\_declaration\_atlas.py}, which writes an index of
  Lean declarations; and
\item \texttt{scripts/build\_publication\_entry\_packet.py}, which writes a
  bounded machine-readable entry point from authored publication metadata.
\end{itemize}

\section{One result from Lean proof to public page}\label{sec:example}

The architecture becomes concrete when one result is followed through every
layer.  We begin with its public meaning, not its internal name:

\begin{quote}
Lean has checked a finite calculation at 28 listed input values.  The open
theorem requires successful values beyond every fixed cutoff.  The 28
calculations therefore do not prove the open theorem.
\end{quote}

This distinction needs no specialist vocabulary.  If a finite list ends at
64, one may choose a cutoff larger than 64.  The open theorem still asks for a
successful value after that cutoff.

\subsection{The formal evidence}

The module
\formalfile{Erdos249257/DiagonalPincerCertificatesT64.lean}
{\texttt{Erdos249257/DiagonalPincerCertificatesT64.lean}}
lists the 28 values and contains a Lean theorem assembling their checked
calculations.  The supported root file \texttt{Erdos249257.lean} imports that
module.  Running \texttt{lake build} asks Lean to check the theorem and every
proof on which it depends.

\subsection{The reviewed public record}

The corresponding object in \texttt{docs/claims.json} contains a public
statement, a result kind, the exact finite domain, five supporting Lean
declarations, and a link to the still-open requirement.  The following is a
shortened view of the real entry; ellipses omit repeated paths and values only
to fit the page.

\begin{verbatim}
{
  "id": "certified_kill_instances",
  "status": "verified finite instance",
  "statement": "Kernel-checked ... at 28 explicit ... scales
                through t = 64 ...",
  "bounded_domain": "the 28 values listed in this record",
  "remaining_open_proposition_ids": [
    "remaining_open.unbounded_certificate_supply"
  ],
  "declarations": [
    "Erdos249257/TotientTailPeriodKiller.lean:404",
    "Erdos249257/LcmDiagonalReduction.lean:237",
    "... three further supporting declarations ..."
  ]
}
\end{verbatim}

The field called \texttt{status} is not a score.  Here ``verified finite
instance'' means only that Lean checked a stated finite set of inputs.  The
field called \texttt{id} is an internal lookup handle for software.  It appears
after the ordinary-language claim because readers should not have to decode it
to understand the result.

\subsection{The public pages and generated routes}

The README and the main mathematical paper state the finite result and the
open boundary.  Generated orientation files provide links and source
locations.  A reader or tool can request the same trace with

\begin{verbatim}
python3 scripts/query_corpus.py --claim certified_kill_instances
\end{verbatim}

The release program then checks several concrete relationships: the five Lean
names occur in the fixed formal source; the source locations remain nearby;
the finite domain is recorded; required public limitation text remains
present; and generated views match a fresh rebuild.  Lean checks the theorem.
The release program checks the recorded description around it.  A person is
still responsible for judging that the description is mathematically faithful.

\section{What happens when a file changes}\label{sec:lifecycle}

For a change to a formal statement or proof, the release path is:

\begin{enumerate}
\item Edit the relevant file under \texttt{Erdos249257/}.
\item Run \texttt{lake build}.  If Lean rejects a proof, stop and repair the
  formal source.
\item Review whether the statement, assumptions, intended meaning, or public
  significance changed.  This is a mathematical judgement.
\item If public meaning changed, update \texttt{docs/claims.json} and follow
  the review requirements in \texttt{docs/methodology.json}.
\item Update the authored README, scope statement, or paper where a reader
  encounters the claim.
\item Run the named builders for generated indexes, source coordinates, and
  paper metadata.
\item Run \texttt{python3 scripts/check\_release.py}.  Any failed recorded
  relationship stops the release.
\item Push the change.  The workflow in \texttt{.github/workflows/lean.yml}
  runs the formal and public-release jobs again in a clean environment.
\end{enumerate}

If only explanatory prose changes, the Lean proof may be untouched, but the
public boundary still matters.  The release check must run.  If the words now
make a stronger or different mathematical claim, the claim record and human
review must change as well.

If a generated file is stale, its builder should be run.  Hand-editing the
output can make the worktree appear fixed while leaving the next rebuild ready
to erase the change.  Builders and generated regions therefore state their
ownership explicitly.

\subsection{What GitHub runs}

The public workflow has two independent jobs.

\begin{itemize}
\item The \textbf{Lean job} checks the dependency lock, installs the pinned
  Lean and Mathlib environment, builds the formal library with a memory-bounded
  planner, completes an ordinary Lake authority check, and builds a small
  downstream example.
\item The \textbf{release-surface job} runs the newcomer checks and the public
  release program.  It then verifies source identity, scope, manuscript and
  licence records, query routes, and deliberately invalid examples for the
  checking programs.
\end{itemize}

Continuous integration repeats commands in a clean checkout.  It does not
perform independent mathematical review and does not turn a consistency check
into proof authority.

\section{Who decides what}\label{sec:trust}

The architecture separates four roles because combining them would make the
guarantee sound stronger than it is.

\begin{table}[H]
\centering\small
\begin{tabular}{@{}>{\raggedright\arraybackslash}p{0.18\textwidth}
                    >{\raggedright\arraybackslash}p{0.34\textwidth}
                    >{\raggedright\arraybackslash}p{0.37\textwidth}@{}}
\toprule
Layer & What it establishes & What it cannot establish \\
\midrule
Lean kernel
& A formal proof has the written formal statement under its imported
  definitions, theorems, and assumptions.
& That the author chose the intended statement or that an English summary is
  faithful. \\
\addlinespace
Human mathematical review
& The formal statement is judged to have the intended meaning; a selected
  public description and its limits are judged appropriate.
& That Lean accepts the proof, or that every important public claim has been
  noticed and recorded. \\
\addlinespace
Release program
& Recorded names, fields, links, generated files, paper identities, and
  selected phrases agree with the declared rules.
& The meaning of arbitrary unrecorded prose, or completeness of the chosen
  comparisons. \\
\addlinespace
GitHub workflow
& Required commands ran in a clean environment and their processes succeeded.
& Independent approval, mathematical truth beyond the formal statements, or
  quality of human review. \\
\bottomrule
\end{tabular}
\caption{The guarantee and non-guarantee of each layer.}
\label{tab:trust}
\end{table}

The repository records review requirements, but it does not technically force
a second independent mathematician to approve every change.  A maintainer can
edit a Lean statement, its public record, and the prose together.  The checks
can show that those files agree with one another; they cannot show that the
maintainer's shared decision was correct.  Review policy and social process
remain outside the automatic trust base.

Similarly, a program-produced fingerprint can prove that two files differ or
match byte for byte.  It cannot determine which differing file expresses the
right mathematics.  The architecture uses automation for repeatable
comparisons and keeps semantic decisions visible as human work.

\section{A claim the checker did not yet cover}\label{sec:failure}

The boundary in Table~\ref{tab:trust} was exposed by one deliberately altered
temporary copy of the repository.  The real README said that 28 checked finite
cases do not provide successful cases beyond every fixed cutoff.  In the test
copy, that wording was changed to say that the finite cases completed the open
requirement.

The formal proofs did not change.  The public-release program also passed,
because nobody had yet listed that particular claim among the phrases and
relationships it must preserve.  The error was outside the written checklist.

The repair had two parts.  First, the boundary was added to the reviewed
public record and required on the appropriate public page.  Second, a small
opposing test case was added: a temporary copy with the required phrase removed
must be rejected.  After the repair, the intact text passed and the same false
edit failed.

This example demonstrates a coverage boundary, not a reliability score.  It
does not show how likely the program is to catch a future error.  It shows
something narrower and more useful for understanding the architecture:

\begin{quote}
The program can preserve a relationship only after a person has identified
and recorded that relationship.
\end{quote}

The detailed historical record is kept separately in
\texttt{docs/publication\_evidence.json}.  That file records the deliberately
altered copies, the original study's limitations, and the fact that original
run logs were not retained.  Internal order labels used by the reconstruction
scripts are bookkeeping for those scripts, not concepts a reader needs in
order to understand this repository.

\section{Operating and adapting the repository}\label{sec:operate}

\subsection{Where a reader should start}

\begin{itemize}
\item For the purpose, file map, and trust boundary, read the root
  \texttt{ARCHITECTURE.md} or this paper.
\item For the mathematical headline and its exact limits, read
  \texttt{README.md} and \texttt{SCOPE.md}.
\item For the mathematical argument, read
  \texttt{erdos249-257-main-paper.pdf}.
\item To follow one result into Lean, use \texttt{docs/SOURCE\_MAP.md} or
  \texttt{scripts/query\_corpus.py}.
\item To change the repository, read \texttt{AGENTS.md} and
  \texttt{CONTRIBUTING.md}, then run the checks owned by the changed files.
\end{itemize}

The public checkout is self-contained.  It is not an interface to a private
agent system, and no public mathematical claim depends on private files.

\subsection{Build dependencies}

The file \texttt{lean-toolchain} pins the Lean version.  The Lake files pin
Mathlib and describe the Lean library.  Python release tools use the standard
library except for named metadata validators installed by continuous
integration.  The paper Makefile uses Tectonic when available and otherwise
uses a standard \LaTeX{} installation.

The principal local commands are:

\begin{verbatim}
# formal proofs
lake build

# newcomer-facing architecture and entry surfaces
python3 scripts/check_architecture_guide.py
python3 scripts/check_cold_clone_comprehension.py --quick

# public claims, documents, generated files, and release rules
python3 scripts/check_release.py

# papers
make -C paper
\end{verbatim}

Each failing command names the file or relationship that did not meet its
recorded rule.  A failure is a stop signal, not evidence that the mathematical
theorem itself is false.  The responsible layer must be inspected: Lean
source, reviewed claim, authored document, generated output, or checking code.

\subsection{What another project could reuse}

Another formal project could reuse the pattern without copying the project's
internal names.  It would need to:

\begin{enumerate}
\item identify the public claims whose strength matters;
\item record, for each selected claim, its ordinary-language meaning, formal
  evidence, finite domain or assumptions, and unresolved conclusion;
\item define which document is authoritative for proofs, public status,
  authored explanation, generated views, and release rules;
\item write comparisons for the relationships that should survive routine
  edits;
\item test each important comparison with both an agreeing and a deliberately
  disagreeing example; and
\item stop publication when any required proof or release check fails.
\end{enumerate}

The data shapes and scripts in this repository are project-local.  The
reusable idea is the separation of authority and the explicit bridge from
formal result to public claim.

\section{Scope and limitations}\label{sec:limits}

This paper documents a public release-engineering subsystem for a Lean
repository.  It is not an architecture for AI agents, model training, or
private research orchestration.  AI systems contributed to development, but
their output has no special authority.  Formal proof authority comes from the
pinned Lean source checked by Lean; public meaning still requires human review.

The architecture does not establish any of the following:

\begin{itemize}
\item a solution of Erd\H{o}s Problem 249 or the universal form of Problem 257;
\item that a formal proposition is the proposition the author intended;
\item that every public sentence has been understood by software;
\item that every important claim has been entered in the reviewed record;
\item that one maintainer's review is independent or adequate;
\item that this project-local design transfers unchanged to another project;
  or
\item that a large number of passing comparisons measures reader
  comprehension, mathematical coverage, or future error-detection probability.
\end{itemize}

The architecture does establish a narrower operational fact: before a public
release, Lean can check the formal proofs, and software can recheck the
specific relationships that people have chosen to record between those proofs
and their public descriptions.

\section{Related systems}\label{sec:related}

Tools such as \texttt{leanblueprint} and LeanArchitect connect proof plans or
exposition to Lean declarations~\cite{leanblueprint,leanarchitect}.  Lean Atlas
helps people inspect whether a formal proposition states the intended
mathematics~\cite{leanatlas}.  Isabelle/DOF assigns checked structure to parts
of formal documents~\cite{isadof}.  Documentation-consistency and mutation
testing research studies how executable systems can be checked against written
descriptions or deliberately faulty inputs~\cite{cascade,demillo,jiaharman}.

This repository combines narrower, ordinary mechanisms: a reviewed JSON
record, links to fixed Lean declarations, generated navigation, exact file
comparisons, known-answer tests, and a blocking release command.  Its purpose
is not to interpret unrestricted prose.  It is to keep selected, reviewed
relationships from drifting silently.

\appendix
\section{A small glossary}\label{app:glossary}

\begin{description}
\item[Lean] A proof assistant that checks formal statements and proofs.
\item[Declaration] Lean's name for a definition, theorem, or other named formal
  object.  Declaration names are lookup handles, not reader-facing concepts.
\item[Claim record] A structured entry describing one selected public claim,
  its evidence, its limits, and any unresolved conclusion.
\item[Generated view] A file or marked file region written by a program from a
  more authoritative source.  It is an index or presentation, not new proof.
\item[Commit] A fixed saved version of the repository.  Formal links use a
  fixed commit so that later edits do not change the theorem being cited.
\item[Digest] A short fingerprint computed from a file's exact bytes.  Equal
  digests support exact identity; they do not judge mathematical meaning.
\item[Continuous integration] Commands run automatically in a clean GitHub
  environment after a push or pull request.
\end{description}

\section{Reproducibility pointers}\label{app:repro}

The reviewed mathematical claims are in \texttt{docs/claims.json}.  The paper
inventory and file identities are in
\texttt{docs/publication\_contract.json}.  The separate historical checker
record is in \texttt{docs/publication\_evidence.json}; executable
reconstructions of its deliberately altered copies are described by
\texttt{experiments/publication\_mutations.json} and checked with

\begin{verbatim}
python3 scripts/run_publication_mutations.py --verify-operators
\end{verbatim}

The original raw outputs for that historical exercise are not registered, and
the reconstruction does not turn replacement targets into original evidence.
The exact formal source cited by the mathematical papers is the saved version
whose full identifier is
\texttt{a16aa24af5236b6508a34b8a4c4b41fdcf620a98}.  These identifiers support
reproduction; they are not concepts needed for first contact.
